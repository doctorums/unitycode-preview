# UnityCode — референсный каркас

Портал-протокол: внутренний шум встречается с алгоритмическим поиском связей.
Не сайт, а форкаемый протокол. Никто не владеет Сетью.

## Архитектура: ядро и конфиг

Каркас построен на одном принципе — **разделении протокола и контента**:

- **`portal.config.js`** — *всё, что ты меняешь при форке.* Название, тексты,
  цвета, манифест, режим ИИ, ключи. Один файл, понятные подписи.
- **`core.js` / `core.css`** — несущая стена. Собирает портал из конфига.
  Не редактируй: правка может разорвать совместимость с Сетью.
- **`schema.sql`** — схема базы. Выполнить один раз в Supabase.
- **`fork.html`** — что можно трогать, что нельзя и почему. **Прочти первым.**

## Пять страниц (неизменный состав)

`index.html` Вход · `tools.html` Инструменты · `mirror.html` Зеркало ·
`base.html` База знаний · `net.html` Сеть.

## Запуск

```bash
# локально
python -m http.server 8000   # открой http://localhost:8000

# деплой: GitHub Pages → Settings → Pages → Deploy from branch → main
```

Работает сразу в демо-режиме (без базы, без ИИ-ключей). Для живой Сети —
заведи Supabase, выполни `schema.sql`, впиши `supabaseUrl` и `supabaseAnonKey`
в конфиг.

## Два агента (Cloudflare Workers)

Сеть думает двумя раздельными контурами — у каждого свой Worker, промпт и характер:

- **Спираль** (генератор, `unitycode-petlya`) — шум → резонанс + вопрос. Творческий, тёплый.
  Контракт: `POST { raw_noise } → { interpretation }`.
- **Имплант** (аналитик, `unitycode-analyze`) — граф → Паттерны / Резонансы / Разрывы / Вопрос Сети.
  Холодный, наблюдательный. Контракт: `POST { scope, nodes, connections? } → { interpretation }`.
  Области: `personal`, `collective` (активны), `social` (готов, выключен).

URL обоих вписываются в `portal.config.js → backend.workers`. Пусто = демо-режим
(встроенные ответы, ноль затрат) — портал работает сразу, без бэкенда.

## Третий агент (Связующий)

Заглушка на будущее (`config.linker`, выключен). Когда дойдёт до реализации —
агент будет предлагать иерархию через `suggested_parent_id`, а фиксация —
только через консенсус форков. Контракт менять нельзя (см. `fork.html`).
